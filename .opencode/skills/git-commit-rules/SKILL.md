## 文件用途

### README.md

项目根目录的 `README.md` 面向**外部使用者**，用于：
- 介绍项目是什么、能做什么
- 快速开始指南（编译、运行）
- 硬件/软件依赖说明
- 项目结构概览

每次新增功能或有重大变更时同步更新 README.md。

### GIT_LOG.md

项目根目录的 `GIT_LOG.md` 面向**开发者自己**，用于：
- 记录每次推送的动机、内容和关键决策
- 保存每次使用的 git 命令，方便日后复用
- 记录遇到的问题和解决方案
- 充当项目开发日志

**每次提交前必须先更新 GIT_LOG.md**，追加本次推送记录。

## Git 提交规则

每次需要 git 提交时，必须按以下顺序执行：

### 第 1 步：检查当前状态

```bash
git status              # 查看工作区状态
git diff --stat         # 查看变更文件概览
git log --oneline -5    # 查看最近提交历史，了解提交风格
```

### 第 2 步：更新 GIT_LOG.md

在 `GIT_LOG.md` 中追加本次推送记录，包含：
- 日期和推送序号（如"2026-05-11 第二次推送"）
- 推送内容说明（改了哪些文件、为什么改）
- 使用的 git 命令示例

### 第 3 步：更新 README.md（如有必要）

如果本次变更影响了项目结构、依赖、用法等外部可见信息，同步更新 `README.md`。

### 第 4 步：更新 .gitignore（如有必要）

检查是否有新的编译产物、临时文件等不应被追踪的文件，及时加入 `.gitignore`。

### 第 5 步：确认暂存内容

```bash
git add <文件列表>
git status              # 再次确认暂存区内容正确
```

### 第 6 步：提交

- 提交信息格式：`<类型>: <简要描述>`
- 类型参考：
  - `init` — 初始提交
  - `feat` — 新增功能
  - `fix` — 修复 bug
  - `chore` — 杂项（构建、配置、工具等）
  - `docs` — 文档变更
  - `refactor` — 重构
  - `perf` — 性能优化
- 描述用中文，简洁说明"为什么改"而不是"改了啥"

### 第 7 步：推送

```bash
git push
```

### 第 8 步：验证

推送完成后，用 `git status` 确认工作区干净，无遗留变更。

## 禁止事项

- 禁止提交编译产物（`.o`、`moc_*`、`Makefile` 自动生成的等），除非用户明确要求
- 禁止提交包含密钥、密码、token 的文件
- 禁止 `--force` 推送到 `main` 分支，除非用户明确要求并知晓风险
- 禁止在未更新 GIT_LOG.md 的情况下直接提交
